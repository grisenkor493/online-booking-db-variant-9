# Проектирование базы данных для языкового центра

Студент: Руслан, группа ИСП-21.

Предметной областью является автоматизация процессов учета, тестирования и записи клиентов в языковой центр. База данных обрабатывает входное тестирование, распределение по уровням и контролирует заполненность групп, формируя очередь при отсутствии мест.

Стек технологий: СУБД MySQL, SQL.

Структура репозитория включает несколько основных разделов. В корне находится файл описания и файл исключений для системы контроля версий. Папка sql содержит основной скрипт развертывания таблиц базы данных. В папке queries располагаются файлы с отдельными аналитическими запросами. Раздел docs отведен под текстовые отчеты в форматах PDF и DOCX, а внутри него есть подпапка images для хранения скриншотов результатов выполнения запросов. Графическая ER-диаграмма вынесена в отдельную директорию diagrams.

Инструкция по развёртыванию базы данных:
1. Склонируйте репозиторий на локальный компьютер.
2. Откройте MySQL Workbench или другую среду управления БД.
3. Выполните скрипт из файла sql/schema.sql для создания структуры таблиц и заполнения их тестовыми данными.

### 🐛 Перечень найденных и исправленных дефектов (Issues)

В ходе независимого тестирования моего проекта напарником было обнаружено и задокументировано несколько дефектов через систему GitHub Issues. Ниже представлен список закрытых задач с техническим описанием их исправления:

#### ❌ Issue #1: Конкурентное (двойное) бронирование одного временного слота
* **Серьёзность:** Критическая (Critical / Security)
* **Описание:** При быстрой отправке формы или при одновременном бронировании времени из двух разных браузеров система создавала две записи на один и тот же час, игнорируя занятость.
* **Как исправлено:** В метод `createAppointment` репозитория `AppointmentRepository` добавлена строгая проверка существования активной записи непосредственно перед выполнением операции `INSERT`. Весь блок обернут в базу данных транзакций PDO (`beginTransaction` / `commit` / `rollBack`) для предотвращения состояния гонки (Race Condition). В случае совпадения времени выбрасывается исключение, и операция безопасно откатывается.
* **Коммит исправления:** `fix: resolve issue #1 — concurrent booking conflict using PDO transactions`

#### ⚠️ Issue #2: Путаница с путями подключения файлов (Fatal error при поиске контроллера)
* **Серьёзность:** Высокая (Major / Bug)
* **Описание:** При переходе в модуль онлайн-записи система выдавала критическую ошибку `Fatal error: Failed opening required` из-за опечатки в `index.php`, где файл `AppointmentController.php` ошибочно запрашивался из папки `repositories/`.
* **Как исправлено:** В главном роутере `index.php` полностью переписан блок подключения внешних зависимостей. Путь для контроллера исправлен на корректный: `require 'controllers/AppointmentController.php';`. Файлы репозитория для стабильности временно интегрированы напрямую, чтобы исключить сбои файловой системы.
* **Коммит исправления:** `fix: resolve issue #2 — fix incorrect require path for AppointmentController`

#### 🎨 Issue #3: Дублирование декларации класса репозитория
* **Серьёзность:** Высокая (Major / Bug)
* **Описание:** На определенных этапах роутинга возникал сбой `Fatal error: Cannot declare class AppointmentRepository`, так как код репозитория дублировался внутри контроллера и главного файла.
* **Как исправлено:** Проведена полная очистка кода (рефакторинг). Из файла `controllers/AppointmentController.php` удалены все лишние объявления классов баз данных. Обязанности строго разделены: `index.php` отвечает за структуры данных, а контроллер занимается исключительно обработкой запросов и подключением `views`.
* **Коммит исправления:** `fix: resolve issue #3 — remove duplicate AppointmentRepository declaration`

Ответ на контрольный вопрос

Уникальный код бронирования в нашей системе формируется с помощью хеширования комбинации данных. Мы используем встроенную функцию md5(), которая принимает на вход строку, состоящую из:

    clientId (уникальный идентификатор клиента)

    courseId (идентификатор выбранного курса)

    datetime (дата и время записи)

    time() (текущая метка времени в секундах для гарантирования уникальности, даже если один и тот же клиент умудрится отправит форму на то же время дважды)

При выводе пользователю в интерфейсе этот длинный MD5-хеш сокращается до первых 8 символов и переводится в верхний регистр для удобства чтения (например, A3F1B2C4).

Использование хеша от динамических параметров здесь надежнее, чем просто случайная строка, так как структура хеша прямо зависит от контекста самой бизнес-операции.
